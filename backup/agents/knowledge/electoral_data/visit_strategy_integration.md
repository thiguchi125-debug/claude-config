# 9地区悉皆訪問 × electoral-district-strategist 統合設計

## なぜ統合が必要か

2つのプロジェクト・エージェントが**同じ「票読み」目的で並走中**：

| 項目 | 🚪9地区悉皆訪問プロジェクト | 🗳️ electoral-district-strategist |
|---|---|---|
| 単位 | **9地区**（市行政区分） | **29投票区**（選挙区分） |
| 目的 | 11,447軒すべての軒先で接点形成 | 投票区別の支持率×投票率で4象限分類 |
| 期間 | 2026/4/27〜9/30 | 月次再評価サイクル |
| 出力 | 訪問記録＋関係性温度 | 4象限マップ＋街宣ROI |
| 未統合だと起きる事故 | 訪問記録が散逸／戦略無視で順序選定／関係性メモ更新漏れ | データ更新源がなく試走分析のまま固定化 |

→ **訪問記録が4象限分析の支持率軸を更新する燃料**になる仕組みが必要。

1軒ごとには記録しない（時間効率破壊）。**地区単位で集計**。

**1日1ページ・地区単位**：
```

地区：[9地区名]

開始-終了：HH:MM-HH:MM
軒数：[訪問軒数] / [当日目標]
内訳：在宅 [X] / 不在 [X] / 配布のみ [X] / 拒否 [X]
連絡先取得：[X]件
LINE登録誘導成功：[X]件
個別「声」（住所をぼかして記録）：
  - [声1]
  - [声2]
  - [声3]
関係性温度の更新：
  - [小エリア・自治会名] → 強支持/支持/中立/要注意
天候・体調・所感：[1行]
```

### デジタル変換（毎日 nichijo 経由）

- nichijo の「記録」モードで上記内容を自動構造化
- 個別の「声」は 📝市民意見リスト DB に転記候補として提示
- 関係性温度の更新は 🗺️地区別関係性メモ DB の月次更新タスクとして蓄積

## 自動化・運用設計

### A. nichijoスキルでの記録フォーマット
nichijoの「記録」モードに以下のテンプレを呼び出せるようにする：
```
記録：訪問 [地区名] [軒数] 在[X]不在[Y]配布[Z]連絡先[N]件
個別の声：[内容]
温度更新：[小エリア]→[温度]
```

→ nichijoの自動分類で：
- 訪問数値は 9地区悉皆訪問プロジェクトページの該当地区に追記
- 個別の声は 市民意見DB に下書き登録
- 温度更新は月次バッチ処理待ち

### B. 月末バッチ処理（推奨：月末タスクとして✅DB登録）
月末ゲート時に手動実行：
1. その月のnichijo訪問ログを集約
2. 関係性メモDB更新（草川承認後）
3. electoral-district-strategist 再起動
4. 4象限分析ページに月次バージョンを保存（履歴管理）

### C. 朝ダッシュボード連動
ohayoスキルが毎朝以下を表示：
- 9地区累計進捗（X軒/11,447軒, X%）
- 当日訪問予定地区（calendarから）
- 4象限分析の最終更新日

## 公選法遵守の二重チェック

訪問・市政報告会・SNS発信の3レイヤ全てに公選法リスクあり：
- **訪問時**：「投票してください」絶対NG。市政報告・意見聴取に徹する
- **報告会時**：参加者の許諾なき録画転載NG、子どもモザイク
- **SNS時**：投票呼びかけ表現NG（事前運動）

→ 月次ゲートで「公選法セルフチェックリスト」も必須。
