**DB役割分担（2026-04-24日次DB統合済み／親ページ `292cf503a68f802da262d7e0dab4ebf3`）**
- **✅ タスクDB** (`collection://292cf503-a68f-81c6-b9dd-000b3ffdd2ce`, page `292cf503a68f814eafcdec9eed56f273`)：全タスクの統合登録先。単発も階層化も同じDBで扱う。`親タスク`／`サブタスク`自己リレーションで階層化、`プロジェクト`でプロジェクトDBへリレーション。相談元氏名・相談元連絡先プロパティあり（市民相談用）。**新規登録は必ず `ステータス: "inbox"`**（その後 Wish List / Done / Archive へ遷移）。タイトルプロパティは `タスク名`、日付は `期限`／`期間`
- **🗂️ プロジェクトDB** (`collection://292cf503-a68f-81fe-bd40-000b64314f2e`, page `292cf503a68f813aa68dd75cbc469c92`)：プロジェクト単位の管理。プロジェクト名／ステータス（バックログ・進行中・完了・中止）／優先度／期間／要約／責任者／`タスク`（タスクDBリレーション）／`進捗率`（タスクのステータスロールアップ）。ボードビュー（ステータス別）あり

**Why:** 2026-04-22にナビマップ統合。2026-04-23に継続DB(814→292c_c)／プロジェクトDB(813)の役割再整理。2026-04-24、日次DB(b43c)と継続DB(292c_c)を統合。理由：単発タスクと階層化タスクをDB単位で分ける必要性が低く、ステータス `inbox` で単発を、`親タスク`＋`サブタスク` で階層化を区別すれば同じDBで運用可能。inboxに全タスクが集まるため優先順位付けが容易になり、エージェント側の分岐ロジックも簡素化された

**How to apply:**
- **すべての新規タスク → ✅ タスクDB（`292cf503-a68f-81c6-b9dd-000b3ffdd2ce`）、ステータス=`inbox` で登録**
- **単発タスク** → 親タスクなしで1件
- **複数ステップ連鎖** → 親タスク＋サブタスクで階層化（`親タスク`リレーションで紐付け）
- **プロジェクト単位の管理が必要** → 🗂️ プロジェクトDBに親プロジェクト作成、`タスク`リレーションで紐付け
- 旧「日次DB」(`b43cf503-a68f-838f-aa23-872cf4b99d92`)はTrash済。参照しているファイルがあれば統合後のDB IDへ置換
- タイトルプロパティ名は `タスク名`（旧日次DBの `名前` ではない）、日付は `date:期限:start`（旧 `date:締切日:start` ではない）
