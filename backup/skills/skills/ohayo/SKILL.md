# おはようスキル — 朝のブリーフィング

ユーザーが朝の挨拶（「おはよう」など）をしたとき、または「今日のタスク」「今日の予定」を聞いたとき、このスキルを実行する。

## やること

以下の3つのデータソースから情報を**並行して**取得し、一気にまとめた朝のブリーフィングを日本語で返す。

### 1. Google Calendar — 今日の予定

`mcp__claude_ai_Google_Calendar__list_events` を使い、今日（現在の日付）の予定を**2つのカレンダーから並行して**取得する。

- calendarId: `t.higuchi125@gmail.com`（プライマリ）と `kusakawa.taku@gmail.com` の2つを同時に取得
- time_min: 今日の00:00:00（ISO 8601形式）
- time_max: 今日の23:59:59（ISO 8601形式）
- 結果をマージして時刻順に並べ、タイトル・開始時刻・終了時刻・場所（あれば）を抽出する

### 2. Gmail — 要チェックメール

`mcp__claude_ai_Gmail__search_threads` を使い、今日届いた未読または重要なメールを取得する。

- query: `is:unread newer_than:1d` または `is:important newer_than:1d`
- max_results: 10
- 件名・送信者・受信日時を抽出し、特に対応が必要そうなものを優先する

### 3. Notion — 朝のダッシュボード＋積み残し確認＋ニュース

以下を**並行して**取得する。

#### 3-0. 📰 ニュースDB — 今朝のニュース

**取得方法（必ずこの順序で実施・分岐を勝手に変えないこと）**：

##### Step A（必須・最初に試す）: notion-query-database-view で Default view を直接叩く

```
view_url: https://www.notion.so/f2eefc669dd54648bbcdacdc8afa1158?v=9d34f9a4-2bc1-4381-a31b-8f8c8ac719e2
```

これでニュースDB全件（直近1週間程度）を一発取得できる。`日付` プロパティで本日（YYYY-MM-DD）のものをローカルでフィルタする。**Default view にはフィルタが無いため、全件返ってくる前提で日付フィルタは取得後に適用すること**。

##### Step B（Step A で日付プロパティを絞り込めなかった場合のみ）: notion-search でDB内検索

```
data_source_url: collection://29e5c1a2-d64d-4822-81fd-0d642c3f07bc
filters: { "created_date_range": { "start_date": "YYYY-MM-DD", "end_date": "YYYY-MM-DD+1" } }
query: "."   ← セマンティック検索を空回りさせる空クエリ。`"ニュース"` 等の単語は禁止（見出しに含まれないため0件になる）
```

##### ⚠️ よくある失敗パターン（過去事故 2026-04-29 踏まえ）

**事故事例（2026-04-29）**：
- `notion-search query="ニュース"` を叩いたが、ニュース見出しに「ニュース」の語がなく**0件返却**
- `data_source_url: collection://f2eefc66-fa30-...` は**database top-level ID**であり data source IDではないため `validation_error: Invalid Data Source URL`
- 結果、cronで本日分8件が正常に収集されていたにも関わらず「未収集」と誤判定

**対策**：
- Step A の view_url は固定値。**変更しないこと**
- collection ID を間違えない: ✅`29e5c1a2-d64d-4822-81fd-0d642c3f07bc`（data source）／❌`f2eefc66-fa30-4f57-...`（database top-level、data_source_url には使えない）
- query パラメータには `"."` または空文字に近いものを使う。**意味のある単語は禁止**

##### 取得後の処理

- `日付` プロパティで本日（YYYY-MM-DD）のものをローカルフィルタ
- ソート: `亀山関連度` 降順、同順なら `カテゴリ` 順
- 取得するプロパティ: `見出し` / `カテゴリ` / `関心テーマ` / `情報源` / `亀山関連度` / `userDefined:URL` / `議会活用メモ`
- 全件抽出（5〜10件想定）

**当日のニュースが本当に0件だった場合**（Step A・B 両方を試した上で確実に0件と確認できた場合のみ）:
- news-briefingスキルをライブ実行する代わりに「⚠️ 本日のニュースは未収集です」と表示し、`/news-briefing` で手動実行できる旨を案内する
- **Step A を試さずに「未収集」と判定するのは禁止**
- フルブリーフィング処理（WebSearch多用）はohayo内では実行しない（重いため別スキルへ委譲する設計）

**⚠️ 重要（2026-04-30事故踏まえ）**: ダッシュボードのタスクテーブルを再利用してはならない。Done/Archive済みのタスクが残っていたり、本日の新規タスクを取りこぼしたりする。**毎朝必ず実DBに最新クエリを投げる**。

```

```

これでタスクDB全件（フィルタなし）を取得できる。**ローカルで以下を必ず実施**：

1. **除外**: `ステータス` が `Done` / `Archive` のタスクは除外（**この除外を忘れると古い完了タスクがダッシュボードに残る**）
2. **包含基準**:
   - **本日期限**: `date:期限:start` = 今日（YYYY-MM-DD）
   - **超過**: `date:期限:start` < 今日 かつ ステータス除外条件後に残っているもの
   - **本日 inbox**: `ステータス` = `inbox` かつ `date:期限:start` = 今日（明示的に拾う）
   - **期限なし inbox はこの一覧では拾わない**（ノイズ防止）— 但し、要約欄の `📋 積み残し` セクションでは inbox 件数だけ集計可
3. **ソート**: `期限` 昇順 → `優先度` 降順（高/中/低/未設定）→ `タスク名`
4. **本日期限と超過の区別**: 表示上、本日期限のタスクは末尾セクション、超過は冒頭セクションに分けて並べる（または「期限」列を追加して一目で分かるようにする）

##### Step B（Step A で has_more=true のとき）: 念のため代表名検索で取りこぼし確認

`notion-search` を使ってダッシュボード旧表に載っていた重要タイトルが現存するか確認：

```
query: "<タスク名の特徴語>"
data_source_url: collection://292cf503-a68f-81c6-b9dd-000b3ffdd2ce
filters: {}
```

##### 取得後の処理（必ず実施）

- ステータス除外を**コードレベルで明示的にフィルタ**（dashboardからのコピペ流用は禁止）
- ⚠️プレフィクス付与: 期限が今日より前のものは表示時に `⚠️ ` を頭に付ける（タイトルへの永続書き込みは不要・表示のみ）
- ★プレフィクス付与: 戸別訪問・市政報告会など特に重要な本日タスクは `⭐` を付与
- 期限列に「本日 4/30」「4/29 (1日超過)」「4/27 (3日超過)」など人間可読な形式で記載
- **件数の恣意的な絞り込みは禁止**。20件超でも全件掲載

##### よくある失敗パターン（2026-04-30事故）

- ❌ `notion-search` を空クエリで叩いて0件 → DB空と誤判定。**必ず view_url 経由の `notion-query-database-view` を使う**
- ❌ ダッシュボードの旧タスクテーブルをそのまま流用 → Done/Archive済みタスクが3件残存し、本日新規タスク14件を取りこぼし
- ❌ 期限超過タスクの ⚠️ をタスク名(DB側)に永続書き込み → 二重マーカー化リスク。**表示時のみ付与**

### 4. Notionダッシュボードへの自動書き込み

**⚠️ この手順は必須。ページが「すでに最新に見える」場合でも必ず実行すること。スキップ禁止。**
**⚠️ チャット出力に表示した重要情報は全てダッシュボードにも反映する。「チャットで言ったから書かない」は禁止。**

ブリーフィング出力後、`mcp__claude_ai_Notion__notion-update-page` で以下の6セクションを一度の `update_content` 呼び出しで更新する。

- page_id: `722beb9e9827421aa5dbbef67c1c4688`
- command: `update_content`

#### old_str の取り方（重要）

**Step 3で取得したページのテキストから、各セクションの先頭行（`## ...`）から次の `##` 行の直前までを正確にコピーして `old_str` に使う。** 自分で推測・生成してはならない。

##### old_str マッチ失敗の罠（2026-04-27事故踏まえ）

`update_content` は `old_str` がマッチしない場合**エラーを返さず黙ってスキップ**する。失敗の典型パターンと対策：

- **特殊文字の取り違え**：「・」(・) と「、」(、) を unicode escape で書くと取り違えやすい。**unicode escape 禁止、平文で書く**
- **長文 old_str はマッチ失敗確率が高い**：1セクション全体を一発で置換せず、見出し+小区切りで複数passに分ける
- **HTMLテーブル・番号付きリストの部分置換は番号体系が壊れる**：必ず`<table>` ... `</table>` 全体を一括置換
- **複数 content_updates を1回で送ると判別不能**：成功・失敗が混在しても返値で判別不可

##### 必須：fetch verify（更新後の検証）

**`update_content` 呼び出し後、必ず1回 `notion-fetch` でページを再取得し、以下をassertする**：
1. `## 🎯 今日のフォーカス（YYYY-MM-DD）` の日付が今日になっているか
2. `## 📅 今日のスケジュール（YYYY年M月D日 曜日）` の日付が今日になっているか
3. `## 📬 重要メール・要対応（YYYY年M月D日更新）` の日付が今日になっているか
4. タスクテーブルの番号が `<td>1</td>` から連番で重複なく並んでいるか
5. 古い箇条書き（前日のフォーカス内容など）が残っていないか
6. **`## ✅ 本日のルーティン` セクションに `- [x] ` が1個も残っていないか**（毎朝リセット必須）
7. **タスクテーブルの全行が Step 3-1b で取得した最新DBクエリ結果と一致しているか**（Done/Archive を含まない・本日期限inboxを含む・件数が新しい・古いタイトルの残骸ゼロ — 2026-04-30事故踏まえ最重要）

**assert失敗時はチャットに警告を表示してユーザーに知らせる。無音スキップは絶対禁止。** マッチ失敗箇所を特定し、平文の短い `old_str` で再試行する。

#### 更新する6セクション（必須）

**セクション1: 今日のスケジュール**

- old_str: ページから取得した `## 📅 今日のスケジュール（...）` セクション全体
- new_str: 以下の形式（Step 1のカレンダーデータで埋める）

```
## 📅 今日のスケジュール（YYYY年M月D日 曜日）
- 終日　イベント名
- HH:MM〜HH:MM　イベント名（場所）
...（取得した全予定を時刻順に）
```

**セクション2: 重要メール**

- old_str: ページから取得した `## 📬 重要メール・要対応` セクション全体
- new_str: 以下の形式（Step 2のメールデータで埋める）

```
## 📬 重要メール・要対応（YYYY年M月D日更新）
- **【送信者】件名** — 対応内容の一行要約
...（広告・定期メルマガ以外の要対応メールのみ）
※上記以外は販促・ニュース配信で返信不要。
```

重要メールが0件の場合: `※本日の要対応メールはありません。` のみを記載。

**セクション2.5: 🎯 今日のフォーカス（毎日新規生成・必須）**

- old_str: ページから取得した `## 🎯 今日のフォーカス` セクション全体（既存があれば）
- new_str: 以下の形式（カレンダー・メール・タスク・積み残しを総合した1日の戦略）

```
## 🎯 今日のフォーカス（YYYY-MM-DD）
**今日の1本軸：[1文で本日の戦略]**
- **午前/午後/夜の動線**：[時間帯別の優先順位と移動経路]
- **本日中に必須**：[期限超過 or 今日中タスクで動かないと致命のもの2〜3件]
- **月曜以降に振替推奨**：[週末で物理的に不可能 or 効率悪いタスク]
- **隙間時間で**：[SNS投稿・写真撮影・ショート動画など]
```

既存セクションがない場合は、`## 📬 重要メール・要対応` の直後に挿入する（old_strに重要メール末尾＋次セクション先頭を含めて挟み込む）。

**セクション2.6: 🗞️ 今朝のニュース（毎日リフレッシュ・必須）**

- old_str: ページから取得した `## 🗞️ 今朝のニュース` セクション全体（既存があれば）
- new_str: 以下の形式（Step 3-0 のニュースDBデータで埋める）

```
## 🗞️ 今朝のニュース（M/D 6:00 JST cron取得・亀山関連度順）
- [カテゴリ][★★★★★] [見出し](記事URL) — 一行コメント（議会活用視点）
- ...（取得した5〜10件全件、亀山関連度降順）
→ 全件は [📰 ニュースDB](https://www.notion.so/f2eefc669dd54648bbcdacdc8afa1158) で確認
```

ニュースDBに本日記事が0件の場合は `⚠️ 本日のニュースは未収集です。/news-briefing で手動取得できます` のみ記載。

**セクション2.7: 🗳️ 選挙進捗ダッシュボード（2026-10-25まで毎日リフレッシュ・必須）**

- old_str: ページから取得した `## 🗳️ 選挙進捗` セクション全体（既存があれば）
- new_str: 以下の形式（Step 3-4b のデータで埋める）

```jsonc

  "new_str": "- [ ] ",
  "replace_all_matches": true
}
```

これでページ上の `[x]` が付いたチェック項目すべてが一発で `[ ]` に戻る（朝・午前〜午後・夜の3層をまとめて処理可）。

- 「## ✅ 本日のルーティン」セクション全体を5項目テンプレで上書きしないこと（既存の3層構造と各リンクが消える）

## 注意事項

- データ取得は4つのソース（Calendar・Gmail・Notion各種・ニュースDB）同時並行で行い、レスポンスを速くする
- ニュース欄はDB読み出しのみ（軽量）。新規取得が必要な場合はnews-briefingスキルへ委譲し、ohayo内でWebSearchを呼ばない
- メールは件名と送信者のみ表示し、本文は含めない（プライバシー配慮）
- Notionのタスクは完了済みのものは除外する
- 「今日のフォーカス」はデータから機械的に出すのではなく、ユーザーの状況を読んで1〜2文で端的にまとめる
- 全体で画面1スクロール以内に収まるよう簡潔にする（ただし「今日のタスク」テーブルは件数にかかわらず全件表示する — 折り畳みや「他◯件」と省略するのは禁止）
- **Notion書き込み（Step 4）は必ず実行する。「ページが最新そう」「同じ内容になる」などの理由でスキップしてはならない**
- **タスク・SNS下書きの数値は必ずDBクエリの結果そのまま出す**。恣意的に絞り込んだり「代表例として2件」という表現で実数と異なる件数を提示するのは禁止
